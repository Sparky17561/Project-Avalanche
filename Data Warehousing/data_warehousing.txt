Fact table display: 

SELECT 
    F.OrderID,
    F.Quantity,
    F.UnitPrice,
    F.TotalAmount,
    DC.DimCustomersKey,
    DP.DimProductsKey,
    DR.DimRegionKey,
    DD.DimDateKey
FROM 
    orderDWH.trans_sales F
LEFT JOIN 
    orderDWH.DimCustomers DC 
    ON F.CustomerID = DC.CustomerID
LEFT JOIN 
    orderDWH.dimproducts DP 
    ON F.ProductID = DP.ProductID
LEFT JOIN 
    orderDWH.DimRegion DR 
    ON DR.Country = F.Country
LEFT JOIN 
    orderDWH.DimDate DD 
    ON F.OrderDate = DD.OrderDate;



Types of FACT TABLES 

Granular/Transactional FACT TABLE -> we dont change the granularity of that table .. 

1 transaction means 1 row .. most commonly used 


 Periodic FACT TABLE -> also known as SnapShot Fact Table 

 1 row != 1 transaction 

 1 row = monthly transaction if snapshot of fact table is done monthly 


Accumulating Fact Table 

1 row = process 

table has so many date / timestamp columns 

every date shows the journey of the transaction .. amazon delivery .. confirmed date , delivered date etc 


Types of DIMENSION TABLES 

Conformed Dimensions 

lets say u have DimProducts and one fact table called FACTORDERS and other being FACTCancelled 

will DimProducts be changed?
no -> one dimension will be shared to multiple facts .. this is called conformed dimensions 


Role Playing Dimensions 

These kind of dimensions are created when fact table with dimension table on multiple conditions/ relnships


so if factOrders and DImDate exist .

and in DimDate there are dates lets say OrderDate and CancelledDate 

thing is the DimDate can perform work of 2 dimensions ... like give only OrderDate or CancelledDate this is called role playing dimensions
